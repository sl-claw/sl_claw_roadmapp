### АКТ 2 — ПРОБЛЕМА (0:30 – 1:30) · 60 сек · что не так с обычным подходом

**🎬 В кадре:** face cam → cut на слайд «3 проблемы продаж в переписке»:
1. Лиды теряются по ночам — 60% клиентов пишут вне рабочего времени
2. Менеджер забывает контекст разговоров недельной давности
3. Пропускная способность одного менеджера — 50 диалогов в день

**🎙 Текст:**
> «Вот три реальные проблемы, с которыми сталкивается любой бизнес, продающий в чатах.
> Первая — лиды теряются по ночам. 60% клиентов пишут вне рабочего времени, и к утру половина уже на сайте конкурента.
> Вторая — менеджер не помнит контекст. Клиент написал на прошлой неделе → сегодня пишет снова → менеджер открывает диалог и спрашивает «как вас зовут?». Это убивает доверие.
> Третья — менеджер физически не может вести больше 50 диалогов в день. Растёт поток заявок — нужно нанимать ещё. И ещё. И ещё.»

**📝 Ремарки:** перечислять чётко, выделять цифры голосом. Каждый пункт по 18-20 сек.

### АКТ 4 — DEMO (2:30 – 5:30) · 180 сек · реальный бот в действии (★ wow-момент)

**🎬 В кадре:** screen share с разделением экрана:
- Слева — Telegram-чат
- Справа — лог сервера (показывает что внутри происходит)

**🎙 Текст:** (прокомментировать каждое действие)

**(2:30 – 3:00):** ведущий печатает «Привет» в TG. Бот отвечает за 2 секунды. В логе видно: `Stage analyzer → 1 (introduction)`.

> «Я печатаю «Привет». Бот отвечает за 2 секунды. Видите справа в логе — он определил стадию «Знакомство». Это первая из семи. Дальше он сам ведёт меня по воронке.»

**(3:00 – 3:45):** ведущий пишет «Я ищу решение для бизнеса». Бот отвечает с уточнениями. Лог: `Stage → 2 (qualification)`.

> «Я говорю что ищу решение. Бот переключился на стадию «Квалификация» — выясняет контекст. Спрашивает по делу: для какого бизнеса, какие задачи. Он не вываливает презентацию сразу.»

**(3:45 – 4:30):** «У меня магазин стройматериалов, поток заявок 200 в день, теряем половину по ночам». Бот выявляет потребность через SPIN. Лог: `Stage → 4 (needs_analysis)`.

> «Я даю контекст. Бот сразу понял проблему — поток лидов больше пропускной способности. Применяет технику SPIN: ситуация → проблема → последствие → выгода. Это методика крупных B2B-продаж, и AI ей владеет.»

**(4:30 – 5:00):** «Сколько стоит?». Бот вызывает `catalog_search` — лог показывает векторный поиск. Возвращает конкретный ответ с ценой.

> «Я спрашиваю цену. Бот идёт в RAG — это поиск по нашему каталогу. Видите в логе — `catalog_search`, 3 результата. Бот отвечает не общими фразами, а с конкретной ценой из нашего прайса.»

**(5:00 – 5:30):** «Хорошо, беру». Бот вызывает `create_payment_link`. Появляется реальная ссылка Stripe. Лог: `Stage → 7 (close)`.

> «Я говорю что беру. Бот генерирует ссылку на оплату через Stripe прямо в чате. Видите — никакого «менеджер пришлёт реквизиты завтра». Сделка закрывается за 90 секунд.»

**📝 Ремарки:** этот акт — главный wow. Печатайте быстро, реакции бота показывайте в полной скорости (не ускорено). Пусть зритель видит реалистичные 2-3 секунды ответа.

### АКТ 3 — КОД (3:00 – 8:00) · 300 сек · построчный разбор

```python
def _build_graph(self):
    graph = StateGraph(SalesState)
    graph.add_node("load_memory", self._load_memory)
    graph.add_node("stage_analyzer", self._stage_analyzer)
    graph.add_node("agent", self._agent)
    graph.add_node("tools", self._tool_node)
    graph.add_node("save_memory", self._save_memory)
    graph.add_node("sync_funnel", self._sync_funnel)

    graph.add_edge(START, "load_memory")
    graph.add_edge(START, "stage_analyzer")
    graph.add_edge("load_memory", "agent")
    graph.add_edge("stage_analyzer", "agent")
    graph.add_conditional_edges(
        "agent",
        self._route_after_agent,
        {"tools": "tools", "save_memory": "save_memory"},
    )
    graph.add_edge("tools", "agent")
    graph.add_edge("save_memory", "sync_funnel")
    graph.add_edge("sync_funnel", END)

    return graph.compile(checkpointer=self.checkpointer)
```

**🎙 Текст:** (читаем построчно)

(3:00 – 4:00) Импорты + создание StateGraph
> «StateGraph принимает на вход тип состояния. SalesState — это TypedDict с полями messages, current_stage, user_id и так далее. LangGraph будет автоматически мержить обновления state между узлами.»

(4:00 – 5:00) add_node — добавляем 6 узлов
> «add_node принимает имя и функцию. Функция получает state, возвращает dict с обновлениями. Простой контракт.»

(5:00 – 6:00) Параллельный fan-out с START
> «Два add_edge от START — load_memory и stage_analyzer. Это параллельный fan-out. LangGraph запустит их одновременно и подождёт оба перед agent. Замер: на нетривиальном сообщении это экономит 800-1500 миллисекунд.»

(6:00 – 7:00) Conditional edge после agent
> «Тут самое интересное. После agent — conditional. Если в последнем сообщении есть tool_calls — идём в tools. Если нет — в save_memory. Это и есть ReAct pattern.»

(7:00 – 8:00) Edge от tools обратно в agent — цикл
> «tools → agent. Это значит после выполнения tool возвращаемся в agent чтобы LLM сформулировал финальный ответ. Может быть несколько итераций — например, agent сделал catalog_search → нашёл → сформулировал ответ.»

```python
async def _stage_analyzer(self, state: SalesState) -> dict[str, Any]:
    last_user = self._last_user_text(state["messages"])

    # Fast path: тривиальные сообщения
    if (
        len(last_user) <= 25
        and self._TRIVIAL_MSG_RE.match(last_user.strip())
    ):
        stage_id = state.get("current_stage", 1)
        logger.info("Stage analyzer FAST: trivial msg %r → keep stage %d",
                    last_user[:30], stage_id)
        return {"current_stage": stage_id, "stage_history": [stage_id]}

    history = self._format_history(state["messages"], limit=10)
    prompt = STAGE_ANALYZER_PROMPT.format(
        stages=stages_overview(state.get("language", "ru")),
        conversation_history=history,
        last_user_message=last_user,
    )
    response = await self.stage_llm.ainvoke([HumanMessage(content=prompt)])
    text = (response.content or "").strip()
    stage_id = int("".join(c for c in text if c.isdigit())[:1] or "1")
    stage_id = max(1, min(7, stage_id))

    logger.info("Stage analyzer -> %d", stage_id)
    return {"current_stage": stage_id, "stage_history": [stage_id]}
```

**🎙 Текст:**
> «Смотрим один узел. _stage_analyzer.
> Берём последнее сообщение пользователя. Если оно короче 25 символов и попадает под регулярку тривиальных фраз — «привет», «спасибо», «ок» — НЕ дёргаем Haiku, оставляем текущую стадию. Это fast-path. Экономит 500-1500 мс на тривиальных сообщениях.
> Иначе — формируем промпт STAGE_ANALYZER_PROMPT, передаём историю последних 10 сообщений, и зовём stage_llm — это Haiku 4.5, дешёвая и быстрая модель.
> Парсим ответ — извлекаем первую цифру. Клипуем в диапазон 1-7. Возвращаем dict с current_stage и stage_history.
> stage_history — массив. LangGraph мержит массивы аппендом, так что history будет накапливаться от вызова к вызову.»

### АКТ 2 — MULTI-QUERY (2:00 – 6:00) · 240 сек

**🎬 В кадре:** код `_llm_expand_query` в rag.py.

```python
async def _llm_expand_query(query: str, settings: Settings) -> list[str]:
    prompt = (
        f"Запрос клиента: «{query}»\n\n"
        "Сгенерируй ровно 2 альтернативных формулировки этого вопроса "
        "для поиска в базе знаний. Каждая — отдельная строка, без "
        "нумерации, без пояснений. Используй разные ключевые слова "
        "и синонимы из той же предметной области, сохраняя смысл."
    )
    llm = ChatAnthropic(model="claude-haiku-4-5-20251001", ...)
    resp = await llm.ainvoke([HumanMessage(content=prompt)])
    variants = [line.strip() for line in resp.content.split('\n') if line.strip()][:2]
    return variants
```

**🎙 Текст:**

(2:00 – 3:00) Концепция
> «Берём запрос пользователя. Просим Haiku — это самая дешёвая модель Anthropic — сгенерировать 2 переформулировки. Получаем 3 вопроса вместо 1.»

(4:00 – 5:00) Демо вызов
> «Запрос: «сколько стоит дом 80 м²». Haiku возвращает: «цена модульный дом 80 квадратов», «прайс дома 80 кв м с террасой». Видите — разные формулировки, тот же смысл. У них будут разные эмбеддинги — поэтому в pgvector найдут разное.»

(5:00 – 6:00) Прирост точности

### АКТ 3 — VECTOR SEARCH + MMR (6:00 – 9:00) · 180 сек

**🎬 В кадре:** код функции поиска в rag.py.

**🎙 Текст:**
